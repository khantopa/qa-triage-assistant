- **Sprint Trigger**: `b30c74a029` by Venkatesh Gopalakrishnan — "SATHREE-41718 - serinity upgrade fixes" (Mar 7, 2026)
  Reordered `deleteAllCookies()` in `AdminLoginSteps.java:goToLoginPageWithUrl()` to run **after** `adminLoginPage.open()` instead of before, to accommodate Serenity 5.x lazy WebDriver initialisation.

- **Root Cause**: Same commit. In Serenity 3.x, `getDriver()` eagerly initialised the WebDriver — calling `deleteAllCookies()` before `open()` deleted cookies on the current (member) domain. In Serenity 5.x, the proxied WebDriver is not instantiated until first page interaction, so `open()` must be called first. This silently changed which domain's cookies get cleared: the call now clears admin domain cookies instead of member domain cookies. The `pcr_hide_after_login=true` cookie set by `skipPcrAfterLogin(true)` during member login is never cleared.

- **Chain of Causation**:
  1. `waitForNewUserLoggedIn(true, true)` during member login calls `skipPcrAfterLogin(true)` → sets `pcr_hide_after_login=true` cookie on member domain (`ProfileWallSteps.java:175-190`)
  2. Test proceeds through email change and liveness flow
  3. `Admin user logs into the application` calls `goToLoginPageWithUrl()` — navigates to admin domain first, then calls `deleteAllCookies()` → only admin domain cookies are cleared; `pcr_hide_after_login` survives on member domain
  4. Test navigates back to `Task - Directly navigate to 'Member' page`
  5. `isAfterLoginPcrEligible()` in `modalRules.ts` reads `ctx.cookies.pcrHideAfterLogin` → `true` → returns `false` → AfterLoginPCR modal never renders → "Complete my profile" button absent → `WaitUntil.elementToBeCurrentlyVisible()` times out

- **Evidence**:
  - `AdminLoginSteps.java:50-58`: `goToLoginPageWithUrl()` — `open()` called before `deleteAllCookies()` since `b30c74a029`
  - `ProfileWallSteps.java:175-190`: `skipPcrAfterLogin(true)` explicitly sets `pcr_hide_after_login=true` cookie on member domain
  - `modalRules.ts:24`: `const showAfterLogin = !ctx.cookies.pcrHideAfterLogin` — cookie gates AfterLoginPCR eligibility
  - `Email_Change.feature:151`: step and feature file unchanged since Sep 2025 (SATHREE-40503) — not a QA trigger
  - `b30c74a029` committed Mar 7, 2026 — 6 days before failing run on Mar 13

- **Pattern Applied**: `admin-session-cookie-survival` (validated this run)

- **Recommended Fix**:
  - `AdminLoginSteps.java:goToLoginPageWithUrl()` — before navigating to admin, explicitly delete the `pcr_hide_after_login` cookie while still on the member domain. Or restore `deleteAllCookies()` before the first `open()` call using `deleteAllCookiesSafely()` (already added to `BasePageObject.java` in `e33bb83055`).
